# Absorb — 외부 지식 흡수 및 시스템 최적화

외부 글/링크에서 Claude Code 활용 팁을 추출하고, 현재 설정과 비교하여 적용 가능한 개선점을 분석한 뒤, Opus 검증으로 시스템 수준 최적화까지 도출한다.

## 입력

`$ARGUMENTS`를 분석하여 모드를 결정한다:

- **URL 모드**: URL이 포함된 경우 → 해당 글에서 팁/기법/설정 추출 후 GAP 분석
- **자체 감사 모드** (인수 없음): 최근 설정 변경 기반 일관성/누락 검사

## 실행 흐름

### Phase 1: 소스 수집 (서브에이전트 위임 — 메인 컨텍스트 보호)

URL이 있으면 **Agent(sonnet)**에 위임하여 글 내용을 수집한다.

서브에이전트 프롬프트:

```
아래 URL의 내용을 읽고, Claude Code / AI 코딩 어시스턴트 활용과 관련된 팁/기법/설정을 추출해줘.

URL: {사용자 URL}

추출 형식:
- **제목**: 글 제목
- **핵심 주제**: 1줄 요약
- **팁 목록**: 각 팁을 아래 형식으로 정리
  1. [팁 이름]: [설명] — [카테고리: workflow/config/prompt/agent/hook/skill]

카테고리 기준:
- workflow: 작업 흐름/프로세스 개선
- config: settings.json, CLAUDE.md 등 설정 변경
- prompt: 프롬프트 작성 기법
- agent: 에이전트 활용/위임 전략
- hook: 훅/자동화 설정
- skill: 스킬/커맨드 활용

응답 형식:
- **결과**: SUCCESS | FAILED
- **핵심 내용**: 위 형식대로 정리
- **미해결 사항**: URL 접근 불가 등 문제 있으면 명시
```

### Phase 2: 현재 설정 탐색 (Explore 에이전트)

**Explore 에이전트**에 위임하여 현재 설정 현황을 수집한다.

```
아래 경로들을 탐색하고 각 파일의 핵심 내용을 1-2줄로 요약해줘:

1. ~/.claude/CLAUDE.md — 글로벌 규칙
2. ~/.claude/commands/*.md — 등록된 명령어 목록 (파일명 + 첫 줄 설명만)
3. ~/.claude/rules/**/*.md — 등록된 규칙 목록 (파일명 + 첫 줄 설명만)
4. .claude/CLAUDE.md — 프로젝트 규칙 (있으면)
5. ~/.claude/settings.json — 훅/권한 설정 (hooks 섹션만)

각 항목을 카테고리별로 정리:
- [workflow] ...
- [config] ...
- [hook] ...
- [skill/command] ...
- [agent] ...
```

> Phase 1과 Phase 2는 **병렬로** 실행한다.

### Phase 3: GAP 분석 (메인에서 수행)

Phase 1의 팁 목록과 Phase 2의 현재 설정을 비교하여:

1. **Already Covered**: 이미 갖추고 있는 것 → 현재 어떤 파일/설정이 커버하는지 명시
2. **Applicable Improvements**: 적용 가능한 개선 → ROI 순으로 정렬
3. **Not Applicable**: 해당 없는 것 → 이유 명시 (기술 스택 불일치, 이미 상위 호환 등)

### Phase 4: Opus 심층 검증 — 시스템 최적화 탐색

Phase 3의 GAP 분석 결과를 **architect 에이전트(opus)**에 전달하여 심층 검증한다.

architect 에이전트 프롬프트:

```
아래는 외부 글의 개선안과 현재 시스템 설정의 GAP 분석 결과이다.
이것을 검증하고 더 깊은 최적화 가능성을 탐색해줘.

## 현재 시스템 개요
{Phase 2에서 수집한 현재 설정 요약}

## GAP 분석 결과
{Phase 3의 Applicable Improvements 목록}

## 검증 요청

1. **개선안 검증**: 각 개선안이 현재 시스템에 실제로 유효한지 판단
   - 이미 상위 호환으로 커버되는 것은 없는가?
   - 현재 아키텍처와 충돌하는 것은 없는가?
   - 적용 시 부작용(side effect)이 예상되는 것은 없는가?

2. **시너지 탐색**: 개선안을 단독 적용이 아닌 기존 시스템과 결합하여 더 큰 효과를 낼 수 있는가?
   - 기존 훅/스킬/규칙과 통합 가능한 패턴
   - 여러 개선안을 묶어 하나의 워크플로우로 만들 수 있는 경우
   - 기존 에이전트 위임 체계에 자연스럽게 녹일 수 있는 방법

3. **업그레이드 제안**: GAP 분석에서 발견하지 못한 추가 최적화 기회
   - 외부 글의 아이디어를 확장하여 현재 시스템에 더 맞게 변형할 수 있는 것
   - 현재 시스템의 약점 중 외부 글의 접근법으로 보완할 수 있는 것
   - 글에는 없지만 같은 철학에서 파생 가능한 개선

응답 형식:
- **검증 결과**: 각 개선안에 대해 ✅ 유효 / ⚠️ 조건부 / ❌ 비추천 판정
- **시너지 기회**: 기존 시스템과 결합 가능한 패턴 목록
- **업그레이드 제안**: 추가 최적화 기회 (ROI 순 정렬)
- **최종 우선순위**: 검증 + 시너지 + 업그레이드를 종합한 추천 액션 순위
```

## 자체 감사 모드 (URL 없을 때)

URL 없이 호출되면 아래를 수행한다:

1. **Explore 에이전트**로 현재 설정 전체 탐색 (Phase 2와 동일)
2. 아래 체크리스트 기반 자체 감사:
   - [ ] CLAUDE.md에 빌드/테스트 명령이 정의되어 있는가?
   - [ ] frozen.yml로 보호해야 할 파일이 등록되어 있는가?
   - [ ] governance.yml로 감시해야 할 변경 패턴이 등록되어 있는가?
   - [ ] 자주 사용하는 워크플로우가 스킬/커맨드로 등록되어 있는가?
   - [ ] 훅이 적절히 설정되어 있는가? (PreToolUse, PostToolUse, Stop)
   - [ ] 에이전트 위임 규칙이 명확한가?
   - [ ] 메모리 시스템이 활용되고 있는가?
   - [ ] 프로젝트별 CLAUDE.md가 존재하는가?
   - [ ] 모듈별 CLAUDE.md가 필요한 모듈에 존재하는가?
3. Phase 3 (GAP 분석)은 체크리스트 결과 기반으로 수행
4. **Phase 4 (Opus 심층 검증)**은 동일하게 실행 — 자체 감사에서도 시스템 최적화 기회 탐색
5. 결과를 동일 형식으로 보고 (Source Summary는 "Self-Audit"로 표시)

## 원칙

- **서브에이전트 위임 필수**: WebFetch, 대량 파일 읽기는 반드시 서브에이전트에 위임 (CLAUDE.md §6)
- **메인은 분석/판단만**: 수집은 서브에이전트, 분석은 메인
- **ROI 기반 정렬**: 적용 난이도 대비 효과가 큰 것을 우선 추천
- **Opus 검증 필수**: GAP 분석만으로 끝내지 않고, 반드시 시스템 수준 최적화 검증을 거친다
- **강제 적용 금지**: 항상 사용자 선택 후 구현 (Plan-First 원칙)
