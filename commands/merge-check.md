현재 브랜치를 베이스 브랜치(develop 또는 main)에 머지해도 안전한지 영향도 분석을 수행해줘.

## 작업 순서

1. 베이스 브랜치 판별: `develop` 브랜치가 있으면 develop, 없으면 `main` 사용
2. `git log {base}..HEAD --oneline`으로 브랜치 커밋 히스토리 확인
3. `git diff {base}...HEAD --stat`으로 변경 파일 통계 확인
4. `git diff {base}...HEAD --name-status`로 추가/수정/삭제 파일 목록 확인
5. `git diff {base}...HEAD`로 실제 변경 내용 상세 확인
6. `git status --short`로 미커밋 파일 확인
7. 프로젝트 빌드 도구를 감지하여 테스트 실행:
   - Gradle: `./gradlew test` (lint 태스크는 제외)
   - Maven: `mvn test`
   - npm/yarn: `npm test` 또는 `yarn test`
8. 아래 분석 프레임워크에 따라 영향도 보고서 작성

## 분석 프레임워크

### 1. 변경 분류
각 변경 파일을 아래 카테고리로 분류:
- **신규 추가**: 기존 코드에 영향 없음
- **기존 수정**: 하위 호환성, 사이드이펙트 확인 필요
- **설정 변경**: 다른 모듈/환경에 전파되는지 확인
- **빌드 변경**: CI/CD 영향 확인

### 2. 영향도 매트릭스
| 영역 | 변경 내용 | 영향 범위 | 위험도 (높/중/낮/없음) |
|------|----------|----------|----------------------|

### 3. 체크리스트
- [ ] 전체 테스트 통과 여부
- [ ] 미커밋 파일 존재 여부
- [ ] 기능 플래그 등으로 신규 기능 격리 여부
- [ ] 공통 모듈 변경 시 다른 모듈 영향
- [ ] 기존 API/인터페이스 하위 호환성
- [ ] 환경별 설정 누락 여부
- [ ] 빌드 의존성 변경 시 CI 영향

### 4. 결론
- **머지 가능** / **머지 보류** / **수정 필요** 중 하나로 판정
- 조건부인 경우 선행 조건 명시

## 규칙

- 추측하지 말고 실제 코드와 설정을 근거로 판단
- 위험도는 "기존 동작이 변경되는가"를 기준으로 평가
- 테스트가 실패하면 원인을 분석하고 수정 방안 제시
- 미커밋 파일이 있으면 반드시 경고
