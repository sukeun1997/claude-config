# Code Trace — 코드 기반 도메인 비즈니스 워크스루

실제 코드를 라인 단위로 따라가면서, 각 단계에서 **어떤 DB 레코드가 생성/변경되고, 어떤 외부 시스템이 호출되는지** 시각화한다. AI 문서 정리가 아닌, **코드를 함께 읽는 능동적 학습**을 지원한다.

## When to Apply

- `/trace <flow명>` 호출 시 (예: `/trace 대출실행`, `/trace 상환`)
- "코드 따라가기", "워크스루", "흐름 따라가줘" 등 요청 시
- 특정 함수나 서비스의 실행 흐름을 이해하고 싶을 때

## 파라미터 파싱

인자에서 flow 식별:
- flow명이 있으면 → `references/flow-registry.md`에서 진입점 조회
- flow명 없이 함수/파일 경로가 있으면 → 해당 함수부터 trace 시작
- 아무것도 없으면 → 등록된 flow 목록 제시

## Phase 1: Flow 진입점 식별

1. `flow-registry.md`에서 해당 flow의 **진입 함수**와 **핵심 코드 경로** 조회
2. 진입 함수 파일을 Read
3. 진입 함수가 호출하는 주요 서비스/태스크 파일들을 병렬 Read (최대 5개)

## Phase 2: Step-by-Step 워크스루

각 단계를 아래 포맷으로 출력:

```
## STEP N: [단계 이름] (파일:라인)

### 코드
실제 코드 스니펫 (핵심 라인만, 10줄 이내)

### 무슨 일이 일어나는가
- DB 변경: [테이블명] × [개수] 생성/수정/삭제
- 외부 호출: [서비스명] (있으면)
- 상태 전이: [필드] 값1 → 값2 (있으면)

### 왜 이렇게 하는가 (추론 or 확인 필요)
- 비즈니스 이유 또는 "🔍 도메인 전문가 확인 필요" 표시

### 실패하면?
- 에러 핸들링 방식, 롤백 범위, 재시작 가능 여부
```

### 워크스루 원칙

1. **코드가 진실**: 문서가 아닌 실제 코드를 Read해서 설명
2. **DB 중심**: 각 단계에서 어떤 테이블의 어떤 레코드가 어떻게 되는지가 핵심
3. **멱등성 표시**: get_or_create 패턴, 재실행 시 동작 차이 명시
4. **외부 의존성 강조**: 외부 API, Kafka, 외부 서비스 호출은 별도 표기
5. **엣지케이스 호출**: 분기 조건(스탁론 전용, 연장 전용 등)을 분기도와 함께 표기

## Phase 3: 요약 + 인터랙티브 Q&A

워크스루 완료 후:

### 3.1 DB 변경 전체 요약
```
이 flow로 생기는 DB 레코드 전체:
┌────────────────────────────────┐
│ 테이블명          × 개수    │
│ ...                        │
└────────────────────────────────┘
```

### 3.2 시나리오 질문 제안
flow에 맞는 "만약 ~하면?" 질문 3-5개를 제안하여 능동적 탐색 유도:

```
질문 던져보세요:
1. "STEP N에서 실패하면 어떻게 돼?"
2. "X 테이블의 Y 필드가 이 값인 이유는?"
3. "이 분기에서 다른 케이스(예: 스탁론)면 어떻게 달라?"
```

### 3.3 도메인 확인 필요 목록
워크스루 중 "왜?"를 코드만으로 설명할 수 없었던 항목 정리:
```
🔍 도메인 전문가 확인 필요:
- [ ] 항목 1
- [ ] 항목 2
```

## Phase 4: 정확성 검증 (Opus Sub-Agent)

워크스루 완료 후, 별도 Opus 서브에이전트로 트레이스 결과의 정확성을 독립 검증한다.

### 검증 에이전트 실행

```
Agent(
  subagent_type: "architect",
  model: "opus",
  prompt: """
  아래 code-trace 워크스루 결과의 정확성을 검증해주세요.

  [워크스루 전체 내용을 전달]

  검증 항목:
  1. **코드 경로 정확성**: 각 STEP의 파일:라인이 실제 코드와 일치하는지
  2. **DB 변경 완전성**: 누락된 DB 변경(save/create/update/delete)이 없는지
  3. **상태 전이 정확성**: 필드 값 변화가 코드 로직과 일치하는지
  4. **호출 체인 누락**: 빠진 서비스 호출이나 Celery task 위임이 없는지
  5. **에러 처리 정확성**: 트랜잭션 범위, 롤백 동작이 실제 코드와 일치하는지

  실제 코드를 직접 Read하여 교차 확인하세요. 추측 금지.

  응답 형식:
  - **결과**: VERIFIED | ISSUES_FOUND
  - **검증 통과 항목**: 정확한 STEP 목록
  - **수정 필요**: 불일치 발견 시 [STEP N] 실제 동작 vs 워크스루 설명
  - **누락 발견**: 워크스루에 빠진 DB 변경/호출/분기
  """
)
```

### 검증 결과 반영

| 검증 결과 | 처리 |
|-----------|------|
| VERIFIED | "✅ Opus 검증 통과" 표시 후 Phase 5로 진행 |
| ISSUES_FOUND | 수정 사항을 워크스루에 인라인 반영 (`⚠️ 검증 수정` 마커) 후 사용자에게 변경 내용 고지 |

### 검증 범위 제한

- 전체 STEP을 재검증하되, 코드 Read는 핵심 분기/DB 변경 지점에 집중
- 검증 에이전트도 동일하게 **추측 금지** — 코드를 Read하지 않은 부분은 검증하지 않음
- 검증은 1회만 수행 (재검증 루프 없음)

## Phase 5: 심화 — 시나리오 트레이스 (선택)

사용자가 질문을 던지면:
1. 해당 분기/함수를 다시 Read
2. 동일한 Step-by-Step 포맷으로 해당 시나리오의 코드 경로를 추적
3. 메인 flow와의 차이점 강조

## 산출물 저장 (선택)

사용자가 요청하면 워크스루 결과를 파일로 저장:
- 경로: `docs/traces/{flow명}.md`
- 포맷: 위 워크스루 전체 + 요약 + 질문 리스트

## 제약사항

1. **추측 금지**: 코드를 Read하지 않은 함수에 대해 설명하지 않음
2. **문서 < 코드**: Notion/기존 문서 내용과 코드가 다르면 코드가 진실
3. **범위 제한**: 하나의 flow를 끝까지 따라간 후 다른 flow로 넘어감 (혼합 금지)
4. **레거시 최소화**: 레거시 동기화 태스크는 존재만 언급하고 상세 trace는 skip (사용자 요청 시만)
